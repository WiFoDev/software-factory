# factory-runtime-v0-0-2 ÔÇö Agent-driven `implementPhase` (single-shot) wired into the default `[implement Ôćĺ validate]` graph

## Intent

Close the human-in-the-loop gap from v0.0.1: the agent does the implementation work autonomously within a single iteration. Adds `implementPhase(opts?)` as a built-in phase that subprocesses out to `claude -p --allowedTools "Read,Edit,Write,Bash" --output-format json` (subscription auth ÔÇö no `ANTHROPIC_API_KEY` required; the original lock also included `--bare`, but post-ship verification showed `--bare` is incompatible with subscription auth in `claude` 2.1+, so it was dropped ÔÇö see Constraints), captures the agent's final `result` text + token counts + files changed + tools used + claude exit status into a new `factory-implement-report` record (`parents: [ctx.runId]`), and enforces a per-phase cost cap (default 100k input tokens) by hard-stopping with `RuntimeError({ code: 'runtime/cost-cap-exceeded' })` after persisting the report. The `factory-runtime run` CLI's default graph becomes `[implement Ôćĺ validate]` (both phases pinned to `cwd: process.cwd()` so agent edits and harness reads land in the same tree); `--no-implement` preserves the v0.0.1 `[validate]`-only behavior. `@wifo/factory-twin` is re-added to `packages/runtime/package.json` and the runtime plumbs `WIFO_TWIN_MODE` + `WIFO_TWIN_RECORDINGS_DIR` env vars to the spawned subprocess so user code can opt into HTTP record/replay through the twin (default mode `'record'`). Iteration auto-loop, cross-iteration record threading, holdout-aware convergence, and `explorePhase`/`planPhase` separation are explicitly deferred to v0.0.3. Public API surface is strictly equal to technical plan ┬ž2 (19 names: 5 functions + 1 class + 13 types).

## Holdout Scenarios

**H-1** ÔÇö Cost-cap path produces both `factory-implement-report` AND `factory-phase` records on disk; validate is **not** invoked
  Given a `[implement Ôćĺ validate]` graph executed via `run({ spec, graph, contextStore, options: { maxIterations: 1 } })`, `implementPhase` configured with `claudePath: <fake-cost-cap>` and `maxPromptTokens: 100`, and `<fake-cost-cap>` reports `usage.input_tokens: 150000`
  When `run` is invoked
  Then the returned `RunReport.status === 'error'`, `iterationCount === 1`; on disk: exactly one `factory-run`, exactly one `factory-implement-report` with `parents: [runId]`, `payload.status === 'error'`, `payload.failureDetail` starts with `cost-cap-exceeded:`; exactly one `factory-phase` record with `payload.phaseName === 'implement'`, `payload.status === 'error'`, `payload.failureDetail` containing `runtime/cost-cap-exceeded`; **zero** `factory-validate-report` records (validate never ran); **zero** `factory-phase` records for `validate`

**H-2** ÔÇö `--no-implement` produces *exactly* the v0.0.1 record set (no implementation-related records)
  Given `factory-runtime run test-fixtures/all-pass.md --no-judge --no-implement --context-dir <tmp>` invoked via `Bun.spawn`
  When the CLI exits
  Then the `<tmp>` directory's record types are *exactly*: 1 `factory-run` + 1 `factory-phase` (with `payload.phaseName === 'validate'`) + 1 `factory-validate-report` = 3 records total; no `factory-implement-report` files exist; the persisted `factory-run.payload.graphPhases === ['validate']` (single-element array, identical to v0.0.1); the stdout summary line is byte-identical to v0.0.1's `factory-runtime: converged in 1 iteration(s) (run=<runId>, <Nms>)`; the CLI runs successfully even when `claude` is not on PATH (verifiable by spawning the CLI with `PATH=/usr/bin:/bin` set in the env, ensuring no `claude` binary is reachable)

**H-3** ÔÇö `is_error: true` causes implement to `'fail'`, but validate **still** runs in the same iteration and the run's final status reflects whichever wins
  Given a `[implement Ôćĺ validate]` graph, `claudePath: <fake-self-fail-but-edit>` (a fake-claude variant that writes `is_error: true` in the JSON envelope BUT also edits `src/needs-impl.ts` with content that satisfies `needs-impl.test.ts`), spec `test-fixtures/needs-impl.md`, and `maxIterations: 1`
  When `run` is invoked
  Then there is exactly one `factory-implement-report` on disk with `payload.status === 'fail'`, `payload.failureDetail` non-empty; there **is** exactly one `factory-validate-report` on disk (validate ran); validate's `payload.status === 'pass'` (the implementation, despite the agent's self-report, satisfied the test); `RunReport.status === 'fail'` because the iteration aggregator sees `['fail', 'pass']` Ôćĺ `'fail'` (it does not promote partial success to convergence); CLI exits 1 with the no-converge summary
  And given the same flow but the fake-claude does NOT edit any file (just writes `is_error: true`), validate's `payload.status === 'fail'` and `RunReport.status === 'fail'` similarly. Both sub-cases prove validate ran (i.e., implement returning `'fail'` does not abort the iteration)

## Constraints / Decisions

- Public API surface from `src/index.ts` is fixed at **19 names**: `run`, `definePhase`, `definePhaseGraph`, `validatePhase`, `implementPhase` (5 functions); `RuntimeError` (1 class); `Phase`, `PhaseContext`, `PhaseResult`, `PhaseGraph`, `PhaseStatus`, `RunOptions`, `RunReport`, `RunStatus`, `PhaseInvocationResult`, `PhaseIterationResult`, `ValidatePhaseOptions`, `ImplementPhaseOptions`, `RuntimeErrorCode` (13 types). Adding a name in v0.0.2 requires updating both this spec and the technical plan ┬ž2. Strict equality.
- `RuntimeErrorCode` gains exactly **three** members in v0.0.2: `'runtime/cost-cap-exceeded'` (post-hoc cost overrun, locked by user), `'runtime/agent-failed'` (coarse operational bucket; specific reason in `failureDetail` prefix), and `'runtime/invalid-max-prompt-tokens'` (factory-call-time validation, symmetric with v0.0.1's `'runtime/invalid-max-iterations'`). The existing six (`graph-empty`, `graph-duplicate-phase`, `graph-unknown-phase`, `graph-cycle`, `invalid-max-iterations`, `io-error`) are unchanged. The set is stable; adding a code is a public-API change. **The public name count stays at 19** because `RuntimeErrorCode` is one type whose union membership grows.
- `implementPhase(opts: ImplementPhaseOptions = {}): Phase`. The produced phase is named `'implement'`. Mirrors the `validatePhase` factory-function pattern. The phase reads `runId` and `iteration` from `PhaseContext` and uses `runId` as the sole parent of the `factory-implement-report` record (`parents: [ctx.runId]`).
- `implementPhase(opts)` validates `opts.maxPromptTokens` synchronously **at factory-call time** (before the closure is constructed). A non-positive integer (`!Number.isInteger(n) || n <= 0`) throws `RuntimeError({ code: 'runtime/invalid-max-prompt-tokens' })` with a `.message` of the form `"must be a positive integer (got '<value>')"`. This is symmetric with how `run()` validates `options.maxIterations`. No record is written.
- `ImplementPhaseOptions` shape (all fields optional):
  - `cwd?: string` ÔÇö default `dirname(spec.raw.filename) ?? process.cwd()`
  - `maxPromptTokens?: number` ÔÇö default `100_000`. Cap is on `usage.input_tokens` from the agent's JSON envelope.
  - `allowedTools?: string` ÔÇö default `'Read,Edit,Write,Bash'`. Forwarded verbatim to `claude -p --allowedTools <value>`.
  - `claudePath?: string` ÔÇö default `'claude'`. Tests pass an explicit path to a fake binary.
  - `timeoutMs?: number` ÔÇö default `600_000` (10 min). Wall-clock; on fire, `SIGKILL` the child and throw `RuntimeError({ code: 'runtime/agent-failed' })` with `agent-timeout` prefix.
  - `twin?: { mode?: TwinMode; recordingsDir?: string } | 'off'` ÔÇö default `{ mode: 'record', recordingsDir: '<cwd>/.factory/twin-recordings' }`. `TwinMode` imported from `@wifo/factory-twin`; not re-exported. The recordings dir is `mkdir -p`'d when mode is `'record'`; left alone when `'replay'`.
  - `promptExtra?: string` ÔÇö appended verbatim under a `# Extra instructions` heading. Optional escape hatch.
- The agent subprocess invocation is `claude -p --allowedTools <list> --output-format json` with the prompt on stdin. No positional argv after `-p`. No `ANTHROPIC_API_KEY` is read or set by the runtime (subscription auth is implicit). **`--bare` was dropped post-ship**: in `claude` 2.1+ the flag strictly disables OAuth/keychain reads, making it incompatible with the locked subscription-auth model. Subscription auth wins (it's the user-facing headline); the rest of the locked surface (`-p`, fixed `--allowedTools` allowlist, `--output-format json`) preserves the reproducibility intent the original lock pursued.
- Cost-cap policy is **post-hoc and hard-stop**: the agent has already consumed the tokens by the time the JSON envelope is parsed. On overrun (`tokens.input > maxPromptTokens`), implementPhase persists the `factory-implement-report` with `status: 'error'` and a `failureDetail` starting with `cost-cap-exceeded: input_tokens=<actual> > maxPromptTokens=<cap>`, then throws `RuntimeError({ code: 'runtime/cost-cap-exceeded' })` with the same detail in `.message`. The runtime catches the throw Ôćĺ `factory-phase` record with `status: 'error'`. The implement-report is **on disk** with `parents: [ctx.runId]` and is discoverable via `factory-context tree <runId>`, even though it is not in `factory-phase.outputRecordIds` (the phase threw before returning the record). Pinned by S-2 / H-1.
- Operational failures (`claude` cannot be spawned / exited non-zero / killed by signal / timeout fired / stdout JSON malformed) all surface as `RuntimeError({ code: 'runtime/agent-failed' })` with a `failureDetail` prefix that names the specific reason. The two run-time codes (`'runtime/cost-cap-exceeded'`, `'runtime/agent-failed'`) cover every implementPhase invocation-time exception path; the third new code (`'runtime/invalid-max-prompt-tokens'`) is factory-call-time. Finer-grained discrimination on agent-failed uses the prefix (`agent-spawn-failed:`, `agent-exit-nonzero (code=N):`, `agent-output-invalid:`, `agent-timeout (after Nms):`, `agent-killed-by-signal SIG:`). No `factory-implement-report` is persisted on operational failure (no envelope to record). Pinned by S-3.
- `is_error: true` in the parsed envelope Ôćĺ `PhaseResult.status === 'fail'` (not `'error'`). The runtime continues phases on `'fail'`, so `validatePhase` runs after `implementPhase` in the same iteration regardless. Validate is the judge of correctness; implement just self-reports task completion. Pinned by S-4 / H-3.
- The `factory-implement-report` schema (zod, in `src/records.ts`):
  - `specId: string`
  - `specPath?: string`
  - `iteration: number` (positive int)
  - `startedAt: string` (ISO-8601 UTC)
  - `durationMs: number` (non-negative int)
  - `cwd: string`
  - `prompt: string` (full prompt as sent ÔÇö includes the spec source as a substring)
  - `allowedTools: string`
  - `claudePath: string`

'')`). **Always populated when the report is persisted**, regardless of `is_error` or cost-cap status; possibly empty string. The audit trail's "what did the agent say it did" question always resolves to this field. Decouples S-5's env-var-plumbing test from `is_error`/`failureDetail` semantics.
  - `filesChanged: { path: string; diff: string }[]` (paths relative to `cwd`)
  - `toolsUsed: string[]` (best-effort)
  - `tokens: { input: number; output: number; cacheCreate?: number; cacheRead?: number; total: number }`
  - `failureDetail?: string` ÔÇö populated only on `status: 'fail'` (mirrors `result` when the agent self-reports failure) or `status: 'error'` (the `cost-cap-exceeded: ÔÇŽ` line; overwrites any prior failureDetail). Independent of `result`.
- File-diff capture: `git diff --no-color HEAD --` if `<cwd>/.git` exists; SHA-256 hash walk fallback otherwise. Pre-state walk uses a **hardcoded** ignore list ÔÇö `node_modules/`, `.git/`, `.factory/`, `dist/`, `build/`, `coverage/`, `.next/`, `.turbo/` ÔÇö and does **not** parse `.gitignore` (node:fs has no native gitignore support and a partial parser is a footgun). Pre-state size cap 5 MB; exceeding it degrades to "paths only" with `diff: ''` per touched file and a one-line `ctx.log` warning. The git path is the v0.0.2 happy path; the hash fallback exists so unit tests against scratch dirs without `.git` still produce non-empty `filesChanged`.
- `toolsUsed` extraction is best-effort: try `parsed.tool_uses` (array of `{ name, ... }` or `string`) and dedup; otherwise infer from disk delta (any new file Ôćĺ `'Write'`; any modified file Ôćĺ `'Edit'`; any stderr line matching `^\$ ` Ôćĺ `'Bash'`). The field's array shape is stable; specific contents are honest about being approximate. The real source of truth is `filesChanged`.
- Token extraction is defensive against missing/non-numeric fields: `numOr0(parsed?.usage?.input_tokens)` etc. Missing field Ôćĺ `0`. The cap check uses `tokens.input` only (the user's "max prompt tokens" framing). If the field is absent across the board, the cap never trips ÔÇö this is documented as a v0.0.2 calibration risk in the README and in the technical plan ┬ž3.

- Sandboxing: the agent runs in the spec's project root cwd. **No git worktree in v0.0.2.** Git-as-undo is the user's safety net; documented in the README and reinforced in the agent prompt's "Constraints" section (`no rm -rf`, no destructive git commands, no installs).
- Cross-iteration record threading is **not** implemented in v0.0.2: the prompt does not include prior iterations' validate-report failures. With `--max-iterations` defaulting to `1` (unchanged from v0.0.1), this is observable only when the user opts into multiple iterations. Documented in README as a v0.0.3 follow-up.
- `--max-iterations` default stays `1`. v0.0.2 is the human-triggered single-shot model per the user's locked scope. v0.0.3 will flip the default once iteration auto-loop + cross-iteration context land.
- Default CLI graph is `definePhaseGraph([implementPhase({ cwd: process.cwd(), ... }), validatePhase({ cwd: process.cwd(), ... })], [['implement', 'validate']])`. **The CLI passes `cwd: process.cwd()` explicitly to both phases** so the agent's edits and the harness's `bun test` invocation resolve against the same tree.
